# 控制策略编辑后需重新发布设计

日期: 2026-05-12
范围: `packages/backend` + `packages/frontend` + `shared/docs`
状态: 已确认设计

## 1. 背景

当前策略模块已经存在“发布”概念，但前后端语义并不一致:

- `SCHEDULE` 自动调度要求 `published_at IS NOT NULL`
- `THRESHOLD` 自动触发当前仍可在未发布状态下执行
- 前端列表未展示“已发布 / 未发布”状态，用户无法区分
- 已发布策略在编辑保存后仍保持已发布状态，容易让用户误以为修改内容一定已经重新审核并生效

这会导致两个问题:

1. 用户无法直观看出哪些策略处于可自动执行状态。
2. “编辑”与“发布”之间缺少明确边界，修改后的策略会直接沿用旧发布状态，操作语义不清晰。

## 2. 目标

本次改造统一控制策略的发布语义，适用于 `THRESHOLD` 与 `SCHEDULE`:

- 新建策略默认处于未发布状态
- 仅已发布策略允许进入自动执行链路
- 策略编辑页内任意一次保存成功后，自动回到未发布状态
- 前端明确展示“已发布 / 未发布”状态
- 前端操作文案与后端实际生效规则保持一致

## 3. 非目标

本次不做以下内容:

- 不引入草稿版本 / 已发布版本双轨模型
- 不按字段粒度判断“哪些变更需要重新发布”
- 不改变手动执行接口的行为
- 不新增数据库字段
- 不将独立条件/目标子资源接口统一纳入“自动回到未发布”范围

## 4. 规则定义

### 4.1 发布状态

策略是否已发布以 `control_policies.published_at` 是否为空为准:

- `published_at IS NOT NULL`: 已发布
- `published_at IS NULL`: 未发布

`published_by` 仅作为发布人审计信息，跟随发布状态一起维护。

### 4.2 自动生效规则

自动执行链路统一要求策略同时满足以下条件:

- `enabled = true`
- `published_at IS NOT NULL`
- 满足各自的触发条件

其中:

- `SCHEDULE`: 到达计划点后执行
- `THRESHOLD`: 收到遥测事件且条件命中后执行

### 4.3 编辑后的状态回退

当用户通过当前策略编辑页完成一次保存时，统一执行以下状态回退:

- `published_at = NULL`
- `published_by = NULL`

这样用户必须重新点击“发布”，策略才重新进入自动执行链路。

本次“编辑页保存”的后端落点为:

- `PUT /api/policies/:id`

原因是当前前端编辑流程会先保存策略主表，再同步条件与目标，因此用户在页面内修改条件或目标，仍会通过主表更新进入“未发布”状态。

本次不扩展到以下独立子资源接口:

- 条件新增 / 修改 / 删除
- 目标新增 / 修改 / 删除

也就是说，本次规则定义为:

- 在当前前端策略编辑页内，只要点保存，就自动回到未发布
- 直接调用独立条件/目标子资源接口时，本次不强制自动回到未发布

该选择是显式产品取舍，而不是遗漏。其影响是:

- 页面内的条件或目标修改会回到未发布
- 绕开页面、直接调用独立子资源接口时，条件或目标变更仍可直接影响自动执行逻辑，且不会强制重新发布

前端文案与后端实现都应明确这是“页面保存后的重新发布规则”，避免将其误解释为“所有策略相关写接口都统一如此”。

### 4.4 发布动作

发布接口保持现有职责:

- 写入新的 `published_at`
- 写入当前操作人 `published_by`

已发布策略再次点击“发布”时，前端应避免重复触发，减少无意义操作。

### 4.5 重新发布后的运行时状态

重新发布后，阈值策略应从全新周期重新参与自动触发评估。

因此发布动作需要同时清理调度器中的运行时内存态，包括:

- 冷却时间 `cooldowns`
- 条件持续命中累计 `condStates`

目标是确保重新发布后不会沿用旧缓存，避免出现以下歧义:

- 策略刚重新发布却被旧冷却时间拦住
- 策略刚重新发布却直接继承旧的持续命中累计

对 `SCHEDULE` 来说，本次仍沿用既有计划点去重逻辑，不在本设计中额外改变 `last_scheduled_for` 的已定义行为。

## 5. 后端设计

### 5.1 `UpdatePolicy()` 语义调整

在当前策略编辑页对应的保存链路中，主表更新成功时统一将:

- `published_at`
- `published_by`

置空。

这项规则同时适用于:

- `THRESHOLD`
- `SCHEDULE`
- 未来若 `DURATION` 落地，也可复用同一发布语义

由于前端当前的保存流程总是先走主表更新，再同步条件与目标，因此无需额外拆分“仅改条件/目标”的页面内保存路径。

### 5.2 调度器发布门槛统一

当前 `SCHEDULE` 已有 `published_at IS NOT NULL` 过滤。

本次需要为 `THRESHOLD` 自动触发也补上同样的过滤条件，确保前后端对“未发布”的理解一致:

- 未发布策略可被查看、编辑、手动执行
- 未发布策略不可自动执行

### 5.3 重新发布时清理运行时缓存

发布接口在成功发布策略后，需要通知调度器清理该策略已有的运行时状态:

- 对 `THRESHOLD`:
  - 清空该策略的冷却时间
  - 清空该策略所有条件的持续命中状态
- 对 `SCHEDULE`:
  - 本次不新增额外清理要求

这样“重新发布”才真正等价于“从新的已发布版本重新开始自动执行”。

### 5.4 调度器依赖落地方案

为了让发布接口能够清理运行时缓存，后端需要将策略调度器实例从“路由注册时临时创建”调整为“可复用依赖”。

推荐最小实现方案:

1. 在 `policy.RegisterRoutes(...)` 中创建单个 `Scheduler` 实例并启动
2. 将该 `Scheduler` 实例注入 policy `Handler`
3. 在 `PublishPolicy()` 成功发布后，调用调度器暴露的按策略清理方法

建议新增的调度器能力:

- `ResetPolicyRuntime(policyID uint64)`
  - 清理指定策略的冷却时间
  - 清理指定策略全部条件的持续命中状态

这样可以避免:

- 重新查询或重建整个调度器
- 在发布接口中直接操作调度器内部 map

该方案保持当前模块边界最小变动:

- handler 只依赖调度器暴露的方法
- 调度器内部状态仍由 `Scheduler` 自己维护

### 5.5 单实例假设

本次设计明确以“单个后端进程内的单个策略调度器实例”为前提。

因此本次运行时缓存清理仅保证:

- 当前进程内一致性

不保证:

- 多实例部署下的跨进程缓存同步

如果未来改为多实例部署，需要进一步引入:

- 分布式锁
- 共享状态存储
- 或发布失效事件广播

本次不扩展到该层级。

### 5.6 可观测性

本次不新增发布状态字段，也不新增审计表。

发布状态的可观测性来自:

- `control_policies.published_at`
- `control_policies.published_by`
- 前端列表展示

如后续需要，可以在审计日志中继续沿用已有 `CREATE_RULE / UPDATE_RULE` 与发布接口日志。

## 6. 前端设计

### 6.1 列表展示

策略列表新增“发布状态”列:

- 已发布: 绿色标签
- 未发布: 橙色或信息标签

可选同时展示发布时间，帮助排查“为何未生效”。

### 6.2 操作按钮

策略操作区需要体现状态差异:

- 未发布策略:
  - 显示可点击的“发布”按钮
- 已发布策略:
  - 显示“已发布”状态文案，或将“发布”按钮置灰禁用

目标是减少重复点击，也让用户明确知道当前策略是否已进入自动执行链路。

同时，发布按钮权限需要与新的工作流闭环一致:

- `ADMIN` 可发布
- `OPERATOR` 也可发布

原因是 `OPERATOR` 当前已经具备策略编辑能力；如果编辑后策略自动回到未发布，则必须同步允许其完成重新发布，避免出现“能改但不能恢复生效”的断链。

### 6.3 提示文案

为了和后端语义一致，前端提交成功提示调整为:

- 新建成功:
  - `策略创建成功，请发布后生效`
- 编辑成功:
  - `策略更新成功，当前为未发布状态，请重新发布后生效`

前端应将提示语义写成“编辑页保存后需重新发布”，而不是“仅主表字段变更后需重新发布”。

## 7. 接口与契约影响

本次不新增 API 字段，使用已有返回字段:

- `published_at`
- `published_by`

但需要明确接口语义:

- 列表与详情接口返回的 `published_at` 是前端判断发布状态的唯一依据
- 更新接口保存成功后，返回结果对应的策略应被视为未发布
- 发布接口是使策略重新进入自动执行链路的唯一入口

同时补齐以下契约:

- `PUT /api/policies/:id` 在当前前端编辑保存链路中承担“回到未发布”的核心职责
- `POST /api/policies/:id/publish` 对未发布策略执行真实发布，并清空该策略的阈值运行时缓存
- 已发布策略再次点击“发布”时，前端应避免重复请求；后端可保留幂等语义，但本次前端需先行规避重复操作

## 8. 测试策略

### 8.1 后端测试

至少补齐以下用例:

- 已发布 `THRESHOLD` 策略更新后，`published_at/published_by` 被清空
- 已发布 `SCHEDULE` 策略更新后，`published_at/published_by` 被清空
- 未发布 `THRESHOLD` 不参与自动触发扫描
- 已发布 `THRESHOLD` 仍可正常自动触发
- 未发布 `SCHEDULE` 不参与定时扫描
- 重新发布 `THRESHOLD` 后，旧冷却时间与条件累计状态被清空
- 当前前端编辑页内修改条件/目标并保存后，策略会变为未发布
- 独立条件/目标子资源接口写操作不会自动清空发布状态
- 发布接口调用后，调度器单实例中的对应运行时缓存已被清理

### 8.2 前端验证

至少验证以下场景:

- 列表正确显示已发布/未发布标签
- 已发布策略的“发布”操作呈现为不可重复点击
- 编辑已发布策略后，刷新列表显示为未发布
- 创建/编辑成功提示文案与新规则一致
- `OPERATOR` 角色可重新发布自己编辑后的策略

## 9. 风险与权衡

### 9.1 用户感知更严格

即使只改名称、优先级等非执行字段，策略在当前编辑页保存后也会回到未发布状态。

这是本次明确选择的产品规则，优点是:

- 用户容易理解
- 后端无需维护复杂的字段差异判断
- 前后端状态一致，不会出现“显示未发布但仍自动执行”的歧义

### 9.2 兼容性影响

历史已发布策略不受迁移影响，只有在后续通过当前编辑页保存后才会自动回到未发布状态。

因此本次无需数据库迁移，也无需批量修正历史数据。

## 10. 实施摘要

本次改造的最小闭环为:

1. 后端 `UpdatePolicy()` 清空 `published_at/published_by`
2. 后端 `THRESHOLD` 自动触发增加 `published_at IS NOT NULL` 过滤
3. 发布接口在重新发布 `THRESHOLD` 时清空运行时缓存
4. 前端策略列表增加发布状态展示
5. 前端操作区按发布状态调整按钮、权限与提示文案
6. 文档与测试同步更新
