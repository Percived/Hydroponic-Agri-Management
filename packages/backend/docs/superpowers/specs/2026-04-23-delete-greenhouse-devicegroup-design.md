# 温室与设备组删除接口设计

## 背景

当前 `internal/device` 模块已具备温室与设备组的创建/更新/查询接口，但缺少删除接口。
本次设计目标是在保持现有风格的前提下补齐删除能力，并确保不存在资源时返回 `404`。

## 设计决策

- 删除语义：沿用当前项目风格，采用物理删除。
- 级联处理：在删除前执行“解绑”而非拒绝删除。
  - 删除温室：先将相关设备 `greenhouse_id` 置空，再将相关设备组 `greenhouse_id` 置空，最后删除温室。
  - 删除设备组：先将相关设备 `group_id` 置空，最后删除设备组。
- 一致性：删除流程使用事务，确保解绑与删除要么全部成功，要么全部回滚。

## API 变更

- 新增 `DELETE /api/devices/greenhouses/:greenhouseId`
- 新增 `DELETE /api/device-groups/:groupId`

返回约定：

- 参数非法：`400`
- 目标不存在：`404`
- 其他数据库错误：`500`
- 删除成功：`200`，`data` 为空对象

## 测试策略

在 `internal/device/handler_test.go` 增加用例：

- 删除温室成功并验证解绑生效
- 重复删除温室返回 `404`
- 删除设备组成功并验证解绑生效
- 重复删除设备组返回 `404`
